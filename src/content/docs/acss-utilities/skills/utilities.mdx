---
title: utilities skill
description: How the utilities skill manages the token source, routes between commands, and controls generation and validation.
---

The `utilities` skill is the omnibus controller for all four `/utility-*` commands. It manages `utilities.tokens.json` (the source of truth), routes to the correct generator or validator, and enforces the atomic write contract.

## Trigger

Triggered by `/utility-add`, `/utility-list`, `/utility-tune`, and `/utility-bridge`. All four commands are thin shells that pass arguments to this skill.

## Step 0 — Exit plan mode

Applies to the write-bearing flows (`/utility-add`, `/utility-tune`, `/utility-bridge`). If the session is in plan mode, the skill calls `ExitPlanMode` before proceeding — those flows shell out to Python scripts and write files, and plan mode blocks all of it.

`/utility-list` is read-only and stays safe under plan mode. The skill only exits plan mode when the user invokes one of the write-bearing flows above.

The skill stays in plan mode only when the user explicitly asked for a preview (for example "show me which families would land", "don't write the bundle yet"). In that case it narrates the file list and bundle size without invoking Write/Edit/Bash and waits for approval.

## The generation pipeline

When `/utility-tune` or `/utility-add` triggers a regeneration:

```
utilities.tokens.json
        ↓
generate_utilities.py
        ↓
assets/utilities/<family>.css  (per-family partials)
        ↓
assets/utilities.css           (concatenated bundle)
        ↓
validate_utilities.py
        ↓
[pass] write to src/styles/    OR   [fail] revert tokens.json
```

## Plan-mode behaviour

When any of `/utility-add`, `/utility-tune`, or `/utility-bridge` triggers a regeneration, exiting plan mode is the first action. `generate_utilities.py` and `validate_utilities.py` both run via Bash, and the final write to `src/styles/utilities.css` is blocked under plan mode. `/utility-list` is read-only and may stay in plan mode. The skill otherwise only stays in plan mode when the user asked for a description without running anything.

## Family concatenation order

The skill enforces a **fixed family order** for stable specificity — the order is always the same regardless of which families are enabled:

```
1. color-bg
2. color-text
3. color-border
4. spacing
5. display
6. flex
7. grid
8. type
9. radius
10. shadow
11. position
12. z-index
```

Color utilities come first (lowest specificity for color). Layout utilities (display, flex, grid) come in the middle. Position and z-index come last.

## CSS class conventions enforced by the skill

**Selector format:** kebab-case, unprefixed:

```css
.bg-primary {
} /* ✓ */
.BgPrimary {
} /* ✗ */
.u-bg-primary {
} /* ✗ */
```

**Responsive format:** hyphen prefix (not colon):

```css
@media (min-width: 48rem) {
  .md-flex {
    display: flex;
  } /* ✓ */
  .md\:flex {
    display: flex;
  } /* ✗ — Tailwind style, requires escaping in JSX */
}
```

**`!important` policy:**

```css
.hide {
  display: none !important;
} /* ✓ — display utilities only */
.sr-only {
  /* ... */
} /* ✓ — visibility utilities only */
.bg-primary {
  background: ...;
} /* ✗ — no !important on color utilities */
.flex {
  display: flex;
} /* ✓ — wait, display utilities get !important */
```

Only `display` and `visibility` families use `!important`. All other utilities rely on source order and normal cascade.

## Token bridge contract

The skill enforces the token bridge parity contract during `/utility-bridge`:

- Every alias defined in `:root` must also be defined in `[data-theme="dark"]`
- Derived variants (`-bg`, `-light`) must be present in both modes
- Missing dark aliases are a validation failure (bridge is not written)

## Bundle size budget

The tokens file includes a `budgetKB` field (default: 15 KB). If `generate_utilities.py` produces a bundle exceeding this limit, the skill warns and asks whether to proceed or trim disabled families further.

## Idempotency check

`validate_utilities.py` runs an idempotency check: it re-runs `generate_utilities.py` and compares the output byte-for-byte to the committed bundle. If they differ, it means someone edited `utilities.css` directly (bypassing the tokens file). The validation fails with a warning to regenerate from `utilities.tokens.json`.
