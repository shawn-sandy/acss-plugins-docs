import { Aside, Steps } from "@astrojs/starlight/components";

Generate fpkit-style React components (`.tsx` + `.scss`) directly into your project. Dependency trees are resolved automatically — requesting `dialog` also generates `button` and `overlay` if they don't exist yet. Pass `--target=html` for static HTML + vanilla JS output instead of React TSX.

## Syntax

```text
/kit-add <component> [component2 component3 ...]
/kit-add --target=html <component> [component2 ...]
```

## Arguments

| Argument        | Required | Description                                                                                          |
| --------------- | -------- | ---------------------------------------------------------------------------------------------------- |
| `component`     | Yes      | One or more component names (space-separated). Case-insensitive.                                     |
| `--target=html` | No       | Generate static HTML + SCSS + vanilla JS instead of React TSX. Defaults to React (`--target=react`). |

## Examples

Generate a single component:

```text
/kit-add button
```

Generate multiple components at once:

```text
/kit-add card badge dialog
```

Generate a complex component (dependencies resolved automatically):

```text
/kit-add table
```

Generate static HTML output for a non-React project:

```text
/kit-add --target=html button card alert
```

## Workflow

When you run `/kit-add <component>`, the command routes to the dedicated `component-<name>` skill for that component (introduced in v1.2.0). Each per-component skill reads its own `reference.md` for templates and writes self-contained TSX + SCSS. The shared steps are:

<Steps>

1. **Init check** — verify `sass` is in `devDependencies` and that `ui.tsx` exists in the target directory. If `ui.tsx` is missing, run the foundation initialization from `kit-core` (see [Foundation CSS install](#foundation-css-install) below).

2. **Lookup** — find the component skill at `skills/component-<name>/SKILL.md`. Normalize the component name to lowercase before resolving the path.

3. **Resolve dependency tree** — read the Generation Contract in the component's `reference.md` to identify all dependencies. For example, `dialog` depends on `button`; `checkbox` depends on `input`.

4. **Show the plan** — display every file that will be created (and any that will be skipped) before writing anything.

5. **Generate bottom-up** — leaf dependencies are written first, then the requested component. This ensures imports resolve correctly from the start.

6. **Handle existing files** — files that already exist are skipped (not overwritten). The existing file's import is used instead.

7. **Display summary** — shows which files were created, which were skipped, and the import statement to add to your code.

8. **Verify integration** — runs `verify_integration.py` against the recorded entry point. Missing imports are surfaced as a numbered fix-up list. The plugin never auto-edits your entry file.

</Steps>

## Foundation CSS install

Starting with `acss-kit` v0.11.0, a vendored `foundation.css` (CSS reset, base typography, and `@layer` cascade ordering) ships alongside `ui.tsx`. The skill applies the following three-case install matrix on first run (Step A4):

| Project state                                                | Action                                                                                                                                      |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| No `ui.tsx` + no `foundation.css` (fresh install)            | Copy both `ui.tsx` and `foundation.css` plus the entire `sass/` source tree; print an import hint                                           |
| `ui.tsx` present, `foundation.css` absent (existing install) | Prompt before copying — explains the CSS reset, `@layer` ordering, and spacing/shadow tokens; you can opt out to avoid the foundation layer |
| Both `ui.tsx` and `foundation.css` present                   | Skip silently — idempotent, nothing to do                                                                                                   |

The `foundation.css` file declares the `@layer` cascade order that all generated components rely on:

```css
@layer foundation, components, utilities, theme;
```

Import `foundation.css` before any component SCSS in your project entrypoint. See [`assets/foundation/SOURCE.md`](https://github.com/shawn-sandy/agentic-acss-plugins/blob/main/plugins/acss-kit/assets/foundation/SOURCE.md) for the upstream pin and the four documented patches (P1–P4) applied to the vendored copy.

## Output

Files are written to `src/components/fpkit/` by default (configurable via `.acss-target.json`).

Each component produces two files:

| File               | Contents                                                           |
| ------------------ | ------------------------------------------------------------------ |
| `<Component>.tsx`  | React component with inline TypeScript types, no external imports  |
| `<Component>.scss` | SCSS with `--comp-*` CSS custom properties and hardcoded fallbacks |

Plus a one-time copy of:

| File     | Contents                                                         |
| -------- | ---------------------------------------------------------------- |
| `ui.tsx` | Polymorphic `UI` base component (copied once, never overwritten) |

## Generation contract

Before writing anything, `/kit-add` shows a preview:

```
Will generate:
  src/components/fpkit/ui.tsx           (foundation — copied once)
  src/components/fpkit/Button.tsx       (new)
  src/components/fpkit/Button.scss      (new)
  src/components/fpkit/Overlay.tsx      (new)
  src/components/fpkit/Overlay.scss     (new)
  src/components/fpkit/Dialog.tsx       (new)   ← requested
  src/components/fpkit/Dialog.scss      (new)   ← requested

Dependencies resolved: button, overlay
```

If a file already exists, the preview shows `(exists — will import)` instead of `(new)`.

## Existing file handling

`/kit-add` **never overwrites** existing files. If `Button.tsx` already exists, the command:

1. Skips generating `Button.tsx` and `Button.scss`
2. Uses the existing file for imports
3. Reports it as "skipped — existing file imported"

This is safe to run repeatedly without risk of clobbering your edits.

## CSS variables

All component styles use `--comp-role-descriptor` naming. Example for `button`:

```css
--btn-bg          /* background color */
--btn-color       /* text color */
--btn-border      /* border color */
--btn-radius      /* border radius */
--btn-padding-x   /* horizontal padding */
--btn-padding-y   /* vertical padding */
```

Override any token in your own CSS — no need to edit the generated file.

## Accessibility features

Every generated component includes:

- `aria-disabled="true"` pattern instead of `disabled` attribute (keeps element focusable and announced)
- `aria-label`, `aria-labelledby`, or `aria-describedby` where applicable
- `:focus-visible` styles using `--color-focus-ring`
- WCAG-appropriate color contrast via theme `--color-*` tokens

## HTML Target (`--target=html`)

Pass `--target=html` to generate static HTML + SCSS + vanilla JS output instead of React TSX — useful for non-React projects, server-rendered apps, static sites, and prototypes.

```text
/kit-add --target=html button card alert
```

### HTML output files

Each component produces up to three files, written to `components/html/` by default (configurable via `.acss-html-target.json`):

| File          | Contents                                                          |
| ------------- | ----------------------------------------------------------------- |
| `<name>.html` | Markup fragment (no `<html>`/`<head>`/`<body>` wrapper)           |
| `<name>.scss` | SCSS — byte-identical to the React generator output               |
| `<name>.js`   | Plain ES module with `init()` function — stateful components only |

On first run, `_stateful.js` is also copied into the target directory. This vanilla-JS helper is required by stateful components (Button, Alert, Dialog).

### HTML-capable components

Only components whose `reference.md` includes an `## HTML Template` section are supported. Run `/kit-list <component>` and look for `HTML output: Verified` to check before generating.

As of acss-kit v1.2.1, the following components support `--target=html`:

| Component | HTML support |
| --------- | ------------ |
| `alert`   | Verified     |
| `button`  | Verified     |
| `card`    | Verified     |
| `dialog`  | Verified     |

All other components (`checkbox`, `field`, `icon`, `icon-button`, `img`, `input`, `link`, `list`, `nav`, `popover`, `table`) are React only — `/kit-add --target=html` will warn when requested.

### HTML Target notes

- The SCSS output is framework-agnostic — identical to the React version.
- The `.js` module is a plain ES module; no bundler required. Import via `<script type="module">`.
- Class names, `data-*` attributes, and ARIA attributes are identical to the React output.
- The target directory is recorded in `.acss-html-target.json` at the project root. Commit this file so teammates write to the same location.

<Aside type="tip">
Run `/kit-list <component>` before `/kit-add` to preview the generation contract, props, CSS tokens, and dependencies for a specific component without generating it.
</Aside>
