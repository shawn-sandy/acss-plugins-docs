Generate fpkit-style React components (`.tsx` + `.scss`) directly into your project. Dependency trees are resolved automatically ‚Äî requesting `dialog` also generates `button` and `overlay` if they don't exist yet.

## Syntax

```text
/kit-add <component> [component2 component3 ...]
```

## Arguments

| Argument    | Required | Description                                                      |
| ----------- | -------- | ---------------------------------------------------------------- |
| `component` | Yes      | One or more component names (space-separated). Case-insensitive. |

## Examples

Generate a single component:

```text
/kit-add button
```

Generate multiple components at once:

```text
/kit-add card badge dialog
```

Generate a complex component (dependencies resolved automatically):

```text
/kit-add table
```

## Workflow

When you run `/kit-add`, the `components` skill executes these steps:

<Steps>

1. **Resolve component name** ‚Äî looks up the component in the reference catalogue and finds its spec doc.

2. **Verify sass and copy `ui.tsx`** ‚Äî checks that `sass` is in `devDependencies`. If missing, stops and tells you how to install it. Copies the polymorphic `UI` base component to your components folder if it doesn't exist already.

3. **Detect build stack** ‚Äî runs `detect_stack.py` to classify framework, bundler, CSS pipeline, and entrypoint. Persists the result into `.acss-target.json` under a `stack` key. If Tailwind is detected, notes that fpkit components and Tailwind utilities coexist without conflict.

4. **Resolve dependency tree** ‚Äî recursively resolves all dependencies. For example, `dialog` ‚Üí `button`, `icon-button`, `icon`; `alert` ‚Üí `icon`.

5. **Show the plan** ‚Äî displays every file that will be created (and any that will be skipped) and waits for confirmation before writing anything.

6. **Generate bottom-up** ‚Äî leaf dependencies are written first, then the requested component. This ensures imports resolve correctly from the start.

7. **Handle existing files** ‚Äî files that already exist are skipped (not overwritten). The existing file's import is used instead.

8. **Display summary** ‚Äî shows which files were created, which were skipped, and the import statement to add to your code.

9. **Verify integration** ‚Äî runs `verify_integration.py` against the recorded entry point. Missing imports are surfaced as a numbered fix-up list. The plugin never auto-edits your entry file.

</Steps>

## Output

Files are written to `src/components/fpkit/` by default (configurable via `.acss-target.json`).

Each component produces two files:

| File               | Contents                                                           |
| ------------------ | ------------------------------------------------------------------ |
| `<Component>.tsx`  | React component with inline TypeScript types, no external imports  |
| `<Component>.scss` | SCSS with `--comp-*` CSS custom properties and hardcoded fallbacks |

Plus a one-time copy of:

| File     | Contents                                                         |
| -------- | ---------------------------------------------------------------- |
| `ui.tsx` | Polymorphic `UI` base component (copied once, never overwritten) |

## Generation contract

Before writing anything, `/kit-add` shows a preview:

```
Will generate:
  src/components/fpkit/ui.tsx           (foundation ‚Äî copied once)
  src/components/fpkit/Button.tsx       (new)
  src/components/fpkit/Button.scss      (new)
  src/components/fpkit/Overlay.tsx      (new)
  src/components/fpkit/Overlay.scss     (new)
  src/components/fpkit/Dialog.tsx       (new)   ‚Üê requested
  src/components/fpkit/Dialog.scss      (new)   ‚Üê requested

Dependencies resolved: button, overlay
```

If a file already exists, the preview shows `(exists ‚Äî will import)` instead of `(new)`.

## Existing file handling

`/kit-add` **never overwrites** existing files. If `Button.tsx` already exists, the command:

1. Skips generating `Button.tsx` and `Button.scss`
2. Uses the existing file for imports
3. Reports it as "skipped ‚Äî existing file imported"

This is safe to run repeatedly without risk of clobbering your edits.

## Available components

Run `/kit-list` for the full catalogue. Quick reference by category:

| Category    | Components                                           |
| ----------- | ---------------------------------------------------- |
| Simple      | `badge`, `icon`, `link`, `divider`                   |
| Interactive | `button`, `checkbox`, `toggle`, `tabs`               |
| Form        | `field`, `input`, `select`, `textarea`, `form-group` |
| Layout      | `card`, `alert`, `banner`                            |
| Complex     | `dialog`, `nav`, `popover`, `table`, `toast`         |

## CSS variables

All component styles use `--comp-role-descriptor` naming. Example for `button`:

```css
--btn-bg          /* background color */
--btn-color       /* text color */
--btn-border      /* border color */
--btn-radius      /* border radius */
--btn-padding-x   /* horizontal padding */
--btn-padding-y   /* vertical padding */
```

Override any token in your own CSS ‚Äî no need to edit the generated file.

## Accessibility features

Every generated component includes:

- `aria-disabled="true"` pattern instead of `disabled` attribute (keeps element focusable and announced)
- `aria-label`, `aria-labelledby`, or `aria-describedby` where applicable
- `:focus-visible` styles using `--color-focus-ring`
- WCAG-appropriate color contrast via theme `--color-*` tokens

<Aside type="tip">
Run `/kit-list <component>` before `/kit-add` to preview the generation contract, props, CSS tokens, and dependencies for a specific component without generating it.
</Aside>
