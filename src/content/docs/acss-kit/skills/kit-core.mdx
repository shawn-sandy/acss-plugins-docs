The `kit-core` skill is the orchestrator for `/kit-create`, `/kit-list`, `/kit-sync`, `/kit-update`, and Form/HTML/Style-Tune modes. Per-component generation was extracted into dedicated `component-<name>` skills in acss-kit v1.2.0.

<Aside type="note">
  In acss-kit v1.2.0, the monolithic `skills/components/` directory was renamed
  to `skills/kit-core/` and scoped to orchestration duties. The fifteen
  per-component skills (`component-alert`, `component-button`, etc.) now handle
  all component generation. See the [Component Skills](#component-skills)
  section below for the full list.
</Aside>

|                     |                                                                                                        |
| ------------------- | ------------------------------------------------------------------------------------------------------ |
| **Triggered by**    | `/kit-create`, `/kit-list`, `/kit-sync`, `/kit-update` plus Form Mode and HTML/Style-Tune mode phrases |
| **Does not handle** | Per-component `/kit-add <name>` requests — those route to `component-<name>` skills directly           |
| **Version**         | `0.4.0`                                                                                                |

## Purpose

`kit-core` generates self-contained, production-quality React components from markdown specs. The developer owns the generated code and can freely modify it. Components use local imports — never `@fpkit/acss`.

## Prerequisites

- React + TypeScript project
- `sass` or `sass-embedded` in `devDependencies`

### Inline-only components

Six components live as inline entries in `kit-core/references/inline-components.md` and are not supported via `/kit-add` as standalone skills: Badge, Tag, Heading, Text/Paragraph, Details, Progress. Promote them via the `/acss-kit-component-author` maintainer skill to create a dedicated per-component skill.

## First-run initialization (Step A)

`kit-core` performs a shared init check that each component skill invokes via `${CLAUDE_PLUGIN_ROOT}/skills/kit-core/SKILL.md`:

1. **Detect project type** — reads `tsconfig.json` and `package.json` to confirm React + TypeScript.
2. **Check sass** — looks for `sass` or `sass-embedded` in `devDependencies`.
3. **Detect package manager** — reads lockfiles (`pnpm-lock.yaml`, `yarn.lock`, `bun.lock`, `package-lock.json`).
4. **Write `.acss-target.json`** — records `componentsDir`, `stack`, and `cssEntryFile`.
5. **Foundation CSS install matrix**:

| Project state                                                | Action                                      |
| ------------------------------------------------------------ | ------------------------------------------- |
| No `ui.tsx` + no `foundation.css` (fresh install)            | Copy both + `sass/` tree; print import hint |
| `ui.tsx` present, `foundation.css` absent (existing install) | Prompt before copying                       |
| Both present                                                 | Skip silently (idempotent)                  |

## Creator Mode

When `/kit-create` is invoked, `kit-core` runs Creator Mode:

1. **Dispatch** — match the component noun against `skills/component-<name>/`. Halt if no match.
2. **Parse** — load the matched `reference.md`, read its Props Interface, and resolve user phrases against the prop set.
3. **Resolve target** — `python3 ${CLAUDE_PLUGIN_ROOT}/scripts/detect_target.py` to locate `componentsDir`.
4. **Vendor** — run `/kit-add <component> [...dependencies]` if any are missing.
5. **Validate** — generic rules plus any per-reference-doc `## Generation Notes — Creator Mode` rules.
6. **Choose mode** — snippet (default) or standalone file.
7. **Generate** — emit JSX with only the props the description resolved.
8. **Summarise** — print resolved spec, deferred sub-components, and a "Refine" prompt.

## Form Mode

Form-shaped requests ("create a signup form with email and password") route to Form Mode inside `kit-core`. Form Mode generates a form component by composing Input, Field, Checkbox, and Button components bottom-up.

## HTML Target Mode

HTML Target Mode generates static HTML + SCSS + vanilla JS components for non-React projects. It is triggered by `/kit-add --target=html`, phrases like "static HTML components" or "HTML version of `<component>`", or when the developer mentions a non-React project.

Both the React and HTML workflows read the same `reference.md` per component. The React workflow extracts `## TSX Template`; HTML Target Mode extracts `## HTML Template` and, for stateful components, `## Vanilla JS`. The `## SCSS Template` block is identical for both.

### Initialization (HT-A)

1. **Target directory** — runs `detect_target.py --target=html` to read `.acss-html-target.json`. On first run, asks where to write HTML components (default: `components/html/`) and writes the config file.
2. **Foundation helper** — copies `assets/html-foundation/_stateful.js` to the HTML components directory if it does not already exist. This file is the disabled-state helper required by stateful components.

### Component generation (HT-B)

1. Looks up `../../component-<name>/reference.md` — same lookup as the React workflow.
2. Reads `## Generation Contract`, `## HTML Template`, `## SCSS Template`, and (for stateful components) `## Vanilla JS`.
3. If `## HTML Template` is missing, warns the developer and offers to hand-author markup from the TSX template. Never silently skips.
4. Resolves dependencies using the same bottom-up algorithm as the React workflow.
5. Shows the dependency tree and waits for confirmation before writing.
6. Generates files bottom-up; skips existing files (the developer owns generated code).

### Output characteristics (HT-C)

| File           | Contents                                                                             |
| -------------- | ------------------------------------------------------------------------------------ |
| `<name>.html`  | HTML fragment — no `<html>`/`<body>` wrapper. Slot placeholders as HTML comments.    |
| `<name>.scss`  | Byte-identical to the React SCSS output — same CSS variables and `data-*` selectors. |
| `<name>.js`    | Vanilla ES module (stateful components only). Exports an idempotent `init()`.        |
| `_stateful.js` | Disabled-state helper — copied once per project.                                     |

Stateful components that emit a `.js` file: Button, Card (interactive variant), Alert, Dialog, Popover, Checkbox, Input, IconButton.

### Verify integration (HT-E)

Runs `verify_integration.py --target=html` after generation. Exit 0 means every `.scss`/`.js` artifact is referenced by at least one page. Exit 1 prints a numbered fix-up list — the skill never auto-edits user pages.

### Key rules (HT-F)

1. HTML fragments only — no `<html>`/`<body>` wrappers.
2. Same class names, `data-*` attributes, and ARIA as the TSX output — SCSS is reused unchanged.
3. Vanilla JS for stateful components — no React, no bundler required.
4. `_stateful.js` is copied once per project and never overwritten.
5. Skip existing files — the developer owns all generated code.
6. Generate in bottom-up dependency order — leaf components first.
7. Never auto-edit user pages — HT-E reports missing references only.

See [`/kit-add --target=html`](/acss-plugins-docs/acss-kit/commands/kit-add#html-target) for the full user-facing command reference.
