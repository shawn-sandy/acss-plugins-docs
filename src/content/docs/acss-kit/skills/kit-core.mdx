`kit-core` is the internal orchestrator that drives `/kit-create`, `/kit-list`, `/kit-sync`, and `/kit-update`. Per-component generation was split out into dedicated `component-<name>` skills in acss-kit v1.2.0. This skill handles shared initialization, Creator Mode, Form Mode, and the read-only inspection workflow.

<Aside type="note">
  As of acss-kit v1.2.0, each component has its own dedicated skill at
  `skills/component-<name>/SKILL.md`. `kit-core` orchestrates across them but
  does not generate component code directly.
</Aside>

## Purpose

`kit-core` provides:

- **Step A ŌĆö First-Run Initialization** shared by all `/kit-add` invocations
- **Creator Mode** ŌĆö natural-language description ŌåÆ component instance (used by `/kit-create`)
- **Form Mode** ŌĆö form-shaped descriptions (e.g. "signup form with email and password")
- **`/kit-list` workflow** ŌĆö read-only inspection of the component catalogue
- **Bulk install workflow** ŌĆö used by `/kit-sync`

## Prerequisites

- React + TypeScript project
- `sass` or `sass-embedded` in `devDependencies`

## Step A ŌĆö First-Run Initialization

Every `/kit-add` invocation starts here.

<Steps>

1. **A1 ŌĆö Detect project type.** Read `tsconfig.json` and `package.json` to confirm React + TypeScript is present.

2. **A2 ŌĆö Check sass.** Look for `sass` or `sass-embedded` in `devDependencies`. If neither is found, halt with an install command.

3. **A3 ŌĆö Detect target dir.** Run `python3 ${CLAUDE_PLUGIN_ROOT}/scripts/detect_target.py <cwd>` to locate or create `.acss-target.json`.

4. **A4 ŌĆö Foundation install matrix.** Apply the three-case matrix for `ui.tsx` and `foundation.css`:

   | Project state                             | Action                                      |
   | ----------------------------------------- | ------------------------------------------- |
   | No `ui.tsx` + no `foundation.css`         | Copy both + `sass/` tree; print import hint |
   | `ui.tsx` present, `foundation.css` absent | Prompt before copying                       |
   | Both present                              | Skip silently                               |

</Steps>

## Step B ŌĆö Component generation (dispatched to per-component skills)

After initialization, `/kit-add` dispatches to the matching `component-<name>` skill. Normalize the component name to lowercase before resolving the path:

```
${CLAUDE_PLUGIN_ROOT}/skills/component-<name>/SKILL.md
```

The per-component skill owns: the canonical TSX/SCSS templates, the Generation Contract, Props Interface, CSS Variables, Accessibility requirements, and dependency resolution.

## Creator Mode

`/kit-create` enters Creator Mode. The workflow:

<Steps>

1. **Dispatch** ŌĆö match the component noun against per-component skills at `skills/component-<name>/`. Halt if no match is found.
2. **Parse** ŌĆö load the matched reference doc, read its Props Interface, and resolve the user's phrases against the prop set (global colour/size synonyms + per-component union literals).
3. **Resolve target** ŌĆö run `detect_target.py` to locate `componentsDir`.
4. **Vendor** ŌĆö run `/kit-add <component> [...dependencies]` if any are not yet present.
5. **Validate** ŌĆö generic rules (empty slots, >80-char content, two-axis conflicts, missing required props) plus any component-specific rules.
6. **Choose mode** ŌĆö snippet (default) or standalone file at `src/components/<Name>.tsx`.
7. **Generate** ŌĆö emit JSX with only the props the description resolved; never silently default a colour-family prop.
8. **Summarise** ŌĆö print the resolved spec, deferred sub-components, and a "Refine" prompt for follow-up tweaks.

</Steps>

**Refinement:** Follow-up phrases like "make it larger", "swap to secondary", "drop the full width" merge into the in-memory spec and re-emit. Say "start over" to clear the spec.

**Out of scope for v0.1:**

- Multi-component compositions in one prompt ŌĆö generate the outer component, then refine.
- Form-shaped descriptions ŌĆö handled by Form Mode.

## `/kit-list` workflow

`/kit-list` is read-only. Component inventory comes from per-component skills only:

- `${CLAUDE_PLUGIN_ROOT}/skills/component-*/SKILL.md` (reads `name:` + `description:` from frontmatter)

| Invocation          | Output                                                                                       |
| ------------------- | -------------------------------------------------------------------------------------------- |
| No arguments        | Categorized listing of all components                                                        |
| With component name | Per-component detail: Generation Contract, Dependencies, Props, CSS Variables, Usage Example |

Lowercase the argument before lookup (e.g. `/kit-list Button` ŌåÆ look for `button`).
