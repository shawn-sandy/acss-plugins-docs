`kit-core` is the internal orchestrator that backs `/kit-create`, `/kit-list`, `/kit-sync`, `/kit-update`, and Form Mode. It was extracted from the earlier `components` skill (v1.2.0) to prevent the LLM router from auto-triggering it whenever a user asks for a component â€” per-component generation now lives in dedicated `component-<name>` skills.

## Purpose

| Triggered by           | Notes                                                       |
| ---------------------- | ----------------------------------------------------------- |
| `/kit-create`          | Creator Mode â€” describe a component in natural language     |
| `/kit-list`            | Read-only catalogue inspection                              |
| `/kit-sync`            | Bulk install all components + foundation + theme            |
| `/kit-update`          | Safe re-copy of unmodified files after an upgrade           |
| `/kit-add <component>` | Delegates immediately to `skills/component-<name>/SKILL.md` |

`kit-core` is **not** the generation engine for individual components. When `/kit-add button` runs, the skill it invokes is `component-button`, not `kit-core`.

## Component inventory

Starting with acss-kit v1.2.0 the component inventory is assembled at runtime by globbing `skills/component-*/SKILL.md` and reading the `name:` and `description:` from each file's frontmatter. There is no static catalog file.

## Creator Mode

Creator Mode (`/kit-create`) turns a natural-language description into a JSX snippet or standalone component file.

<Steps>

1. **Dispatch** â€” match the component noun in the user's description against available per-component skills at `skills/component-<name>/`. Halt with a list of known components if no match is found.

2. **Parse** â€” load the matched component's `reference.md`, read its Props Interface, and resolve user phrases against the prop set. Global synonym tables (colour family, size family, variant synonyms) are applied before per-component rules.

3. **Resolve target** â€” run `python3 ${CLAUDE_PLUGIN_ROOT}/scripts/detect_target.py` to locate the `componentsDir`.

4. **Vendor dependencies** â€” run `/kit-add <component> [...dependencies]` for any component not yet present in the project.

5. **Validate** â€” apply generic rules (empty slots, content over 80 chars, two-axis conflicts, missing required props) plus any per-component Generation Notes the reference doc declares.

6. **Choose output mode** â€” snippet (default) or standalone file at `src/components/<Name>.tsx`.

7. **Generate** â€” emit JSX using only the resolved props; never silently default a colour-family prop.

8. **Summarise** â€” print the resolved spec, any deferred sub-components, and a "Refine" prompt for follow-up tweaks.

</Steps>

**Refinement:** follow-up phrases like "make it larger", "swap to secondary", "drop the full width", or "change the title to 'â€¦'" merge into the in-memory spec and re-emit. Say "start over" to clear the spec.

## `/kit-list` workflow

Read-only catalogue inspection â€” never writes files.

<Steps>

1. **No arguments** â€” glob `skills/component-*/SKILL.md`, read `name:` + `description:` from each frontmatter, and print a categorized listing. Components with a Verified HTML entry are marked `[HTML]`.

2. **With component name** â€” locate `skills/component-<name>/SKILL.md`, load `reference.md`, and output: Generation Contract, Dependencies, HTML output status, Props, CSS Variables, and a Usage Example.

3. **Unknown name** â€” list the closest matches (fuzzy name match) and stop.

</Steps>

## CSS variable conventions

All component tokens follow the `--comp-role-descriptor` pattern:

| Segment      | Meaning                | Example                                |
| ------------ | ---------------------- | -------------------------------------- |
| `comp`       | Component abbreviation | `btn`, `card`, `dlg`                   |
| `role`       | Semantic role          | `bg`, `color`, `border`, `radius`      |
| `descriptor` | Optional modifier      | `hover`, `focus`, `active`, `disabled` |

Approved abbreviations: `bg` â†’ background, `bd` / `border` â†’ border, `rad` / `radius` â†’ border-radius, `clr` / `color` â†’ text color, `sz` / `size` â†’ font-size, `fw` â†’ font-weight, `px` / `py` â†’ horizontal / vertical padding, `mx` / `my` â†’ horizontal / vertical margin, `el` â†’ elevation / shadow.

## `aria-disabled` pattern

Generated components use `aria-disabled="true"` instead of the `disabled` attribute. This preserves keyboard focusability and screen-reader announcement while appearing disabled visually. The disabled visual state is styled via `[aria-disabled="true"]` in the component's SCSS.
