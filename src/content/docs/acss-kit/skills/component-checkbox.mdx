The `component-checkbox` skill generates a self-contained accessible Checkbox component directly into a developer's project. It is invoked via `/kit-add checkbox` or `/kit-create` (then ask for a checkbox).

<Aside type="note">
  This skill was added in acss-kit v1.2.0. It sets `disable-model-invocation:
  true` so it stays out of session auto-context. Invoke it explicitly via
  `/kit-add checkbox`, `/kit-create`, or by calling `component-checkbox` by
  name.
</Aside>

|                  |                                                                                                                                   |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **Triggered by** | `/kit-add checkbox`, `/kit-create` (checkbox description), direct `component-checkbox` skill name                                 |
| **Output**       | `checkbox.tsx` + `checkbox.module.scss`                                                                                           |
| **Dependencies** | `input` â€” generates Input first if missing                                                                                        |
| **Hint**         | Describe the label, the initial state (checked/unchecked/indeterminate), controlled vs uncontrolled, and any error/required state |

## Workflow

<Steps>

1. **Read** `reference.md` in the skill directory for the canonical TSX/SCSS templates, Generation Contract, Props Interface, CSS Variables, Accessibility requirements, and Usage Examples.

2. **Init check** â€” run `python3 ${CLAUDE_PLUGIN_ROOT}/scripts/detect_target.py <cwd>`. If the foundation (`ui.tsx`) is missing, run Step A of `${CLAUDE_PLUGIN_ROOT}/skills/kit-core/SKILL.md` to initialize it before generating this component.

3. **Dependencies** â€” Checkbox depends on `input`. Before generating Checkbox, check whether `<targetDir>/input.tsx` already exists. If missing, invoke the `component-input` skill first, then continue.

4. **Generate** â€” apply the Generation Contract from `reference.md` to produce:
   - `<targetDir>/checkbox.tsx` from the `## TSX Template` section
   - `<targetDir>/checkbox.module.scss` from the `## SCSS Template` section

5. **Verify** â€” run `python3 ${CLAUDE_PLUGIN_ROOT}/scripts/verify_integration.py <projectRoot>` and print the summary.

</Steps>

## Design tokens (v1.11.0)

Token references in `checkbox.component.md` now use the DESIGN.md/Material-3 vocabulary so they resolve against a `DESIGN.md` produced by `/design-export`.

| Token path                 | Role                                                            |
| -------------------------- | --------------------------------------------------------------- |
| `{colors.surface}`         | Checkbox background                                             |
| `{colors.outline-variant}` | Border color (was `{colors.border}` before v1.11.0)             |
| `{colors.primary}`         | Checked background and border                                   |
| `{colors.on-surface}`      | Label color (was `{colors.text}` before v1.11.0)                |
| `{colors.error}`           | Required indicator color (was `{colors.danger}` before v1.11.0) |
| `{rounded.sm}`             | Border radius                                                   |
| `{spacing.sm}`             | Gap between checkbox and label                                  |
| `{typography.body-md}`     | Label typography                                                |

The emitted CSS still uses `var(--color-*)` custom properties â€” `/kit-add` output is byte-identical. Only the COMPONENT.md front-matter `tokens:` references changed.

## Accessibility

- `aria-checked` supports `true`, `false`, and `"mixed"` for the indeterminate state.
- Custom indicator overlays the native checkbox (visually hidden via CSS) so the native element stays in the accessibility tree.
- `aria-invalid` and `aria-required` wired for form validation.
